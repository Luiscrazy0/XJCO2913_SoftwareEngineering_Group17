# 业务逻辑与实现约束的冲突分析

## 文档信息

| 字段 | 内容 |
|------|------|
| 创建日期 | 2026-04-27 |
| 触发背景 | Sprint 4 规划中否决自由流动方案，回归 Station 模式 |

---

## 1. 核心矛盾

本项目名为「共享电动滑板车租借系统」，但与真实共享滑板车业务存在以下不可调和的冲突：

> **真实世界的滑板车共享 = 自由流动（Free-Floating），但我们的技术边界只支持站点取还（Station-based）。**

---

## 2. 逐项冲突分析

### 2.1 车辆定位与寻车

| 维度 | 真实业务要求 | 当前实现能力 | 冲突 |
|------|-------------|-------------|------|
| 车辆发现 | 打开 APP，地图上显示附近每辆滑板车的实时位置 | Scooter 有 `latitude`/`longitude` 字段，但无 IoT 通信更新，坐标为静态数据 | 地图上若标出每辆车的位置，用户走到坐标点找不到车（坐标可能过时；即使准确，用户也无法区分哪辆是自己的） |
| 导航到车 | APP 内步行导航到车辆 GPS 位置 | 高德地图可做导航，但寻车的前提是坐标真实且车辆有外观标识 | 无物理设备，坐标只是数据库中的数字 |
| 鸣笛寻车 | 点击「鸣笛」→ IoT 指令让车辆发出声音 | 无 IoT | 不可实现 |

**当前决策：** 不在地图显示单辆车。以 Station 为唯一地图标记，用户凭常识到达站点（图书馆、食堂等物理地标），到站后根据车辆编号找到预定车辆。

---

### 2.2 解锁 / 开始使用

| 维度 | 真实业务要求 | 当前实现能力 | 冲突 |
|------|-------------|-------------|------|
| 解锁方式 | 扫描车身 QR 码 → IoT 开锁 → 物理锁打开 | Web 端无摄像头 API（即便是移动端浏览器也受限）；无 IoT | Web 应用无法扫描二维码，也无法向硬件发送指令 |
| 解锁验证 | 只有预订用户本人能解锁（蓝牙 + Token） | 无设备认证 | 无法防止非预订用户取走车辆 |
| 物理交接 | 用户自行取车 | 需要站点管理员或自助取车流程 | `startRide` 只是数据库状态变更，不涉及物理动作 |

**当前决策：** 放弃自动解锁。用户预订后获得确认信息，到达站点后手动点击「开始骑行」确认已取到车。依赖站点管理员的物理看管（或诚信机制）。

---

### 2.3 骑行中追踪

| 维度 | 真实业务要求 | 当前实现能力 | 冲突 |
|------|-------------|-------------|------|
| 实时位置追踪 | APP 持续上传用户 GPS + 车辆 GPS（双源校验） | 无车辆 IoT；浏览器 `navigator.geolocation` 不可靠（后台时暂停） | 无法判定用户是否正在骑行、骑了多远、是否偏离服务区 |
| 电子围栏 | 离开服务区 → 告警 + 远程限速；进入禁停区 → 无法锁车 | 无 IoT 远程控制 | 地理围栏无法执行 |
| 摔倒检测 | IMU 检测倾角突变 → 自动触发紧急联络 | 无传感器 | 不可实现 |
| 电量监控 | 实时电量上报 → 低电量车辆自动下线 | 无电池通信 | 不可实现 |

**当前决策：** 骑行期间仅做计时，不做位置追踪和地理围栏。`IN_PROGRESS` 状态下仅显示计时器和费用。不尝试模拟 GPS 追踪。

---

### 2.4 还车 / 结束使用

| 维度 | 真实业务要求 | 当前实现能力 | 冲突 |
|------|-------------|-------------|------|
| 还车验证 | GPS 验证（在服务区内）+ 停车照片（AI 识别是否规范停放） | 浏览器可获取位置、可拍照，但无 AI 识别 | 位置可获取但不精确；照片可拍但无人审核 |
| 锁车 | APP 点击「结束」→ IoT 指令锁车 → 物理锁闭合 | 无 IoT | 锁车不可控 |
| 费用校准 | 按实际骑行时长（解锁→锁车）精确计费，超时自动续费 | 只能按预订时长计费；无解锁/锁车时间戳 | 用户可能提前还车（多付钱）或延迟还车（不补费） |
| 不在站点还车 | 用户停在非站点区域 → 收调度费 | 无 GPS 追踪 | 无法判定用户是否真的还到了站点 |

**当前决策：** 
- 用户手动选择还车站点并确认。依赖诚信机制。
- 按预订时长计费（非实际时长）。后续可考虑按 `actualStartTime` → `actualEndTime` 差额重新计算，但目前优先保证闭环可用。
- 还车时要求用户确认「车辆完好」，若不勾选 → 自动生成 DAMAGE Feedback。

---

### 2.5 支付与计费

| 维度 | 真实业务要求 | 当前实现能力 | 冲突 |
|------|-------------|-------------|------|
| 支付网关 | 对接 Stripe / 支付宝 / 微信支付，真实扣款 | 无商户账号，无法对接真实支付 | 只能模拟支付（数据库直接标记 SUCCESS） |
| 预授权 | 租车前预授权押金（如 ¥200），还车后按实际费用扣款，释放余额 | 无支付网关 | 无法做预授权 |
| 超时扣费 | 超时自动续费 | 无定时任务/消息队列 | 超时不会自动触发扣费 |
| 退款 | 取消预订退全款；提前还车退差额 | 模拟支付无真实资金流 | 退款只是数据库操作 |

**当前决策：** 支付模拟为数据库操作。计费采用预订时锁定价格模式（创建 Booking 时确定 totalCost），不做按实际时长的动态计价（避免引入计费 bug）。

---

### 2.6 平台形态限制

| 维度 | 真实业务要求 | 当前实现能力 | 冲突 |
|------|-------------|-------------|------|
| 客户端 | 原生 APP（iOS/Android），可使用蓝牙、摄像头、后台定位、推送通知 | Web 应用（React + Vite）| 无蓝牙、后台定位不可靠、无推送、Web 摄像头 API 兼容性差 |
| 实时通信 | WebSocket / MQTT 长连接（用于 IoT 指令和实时状态同步） | 纯 REST API（HTTP 请求-响应） | 无服务端推送能力；状态更新靠轮询 |
| 离线能力 | APP 本地缓存，弱网环境仍可使用 | Web 应用离线能力有限（Service Worker / PWA 可做但不在此范围） | 弱网环境功能基本不可用 |

**当前决策：** 接受 Web 平台限制。不做离线支持。状态同步靠前端定时轮询或用户手动刷新。

## 3. 已确认的妥协清单

以下功能因技术边界限制，在本项目中**明确不做**：

| # | 功能 | 原因 |
|---|------|------|
| 1 | 自由流动取还（Free-Floating） | 无 GPS 寻车、无 IoT 解锁 |
| 2 | QR 码扫描解锁 | 无摄像头集成、无 IoT |
| 3 | 实时 GPS 位置追踪 | 无车辆 IoT |
| 4 | 电子围栏（Geofencing） | 无 IoT 远程控制 |
| 5 | 电池电量实时监控 | 无电池通信模块 |
| 6 | 远程锁车 / 限速 | 无 IoT |
| 7 | 真实支付网关对接 | 无商户账号 |
| 8 | 按实际骑行时长动态计费 | 无可靠解锁/锁车时间戳 |
| 9 | 蓝牙近场通信 | Web 平台不支持 |
| 10 | 原生 APP 推送通知 | Web 平台，非 APP |
| 11 | OTA 固件升级 | 无物理设备 |

## 4. 已确认的折中方案

| # | 原始需求 | 折中方案 | 风险 |
|---|---------|---------|------|
| 1 | 地图上看到每辆滑板车 | 地图只显示 Station，点击 Station 展开车辆列表 | 用户无法直观看到车辆分布密度 |
| 2 | 扫 QR 码解锁 | 预订后获得确认信息，到站点后手动点击「开始骑行」 | 依赖诚信；无法防止冒领 |
| 3 | 按实际时长计费 | 预订时锁定价格（按 `hireType` 固定价格） | 提前还车不退、超时还车不补 |
| 4 | IoT 实时位置 | 无。Scooter 坐标仅在做数据库操作时更新（取车/还车时） | 无法追踪车辆在骑行中的位置 |
| 5 | 自动结束行程 | 用户手动点击「结束骑行」+ 选择还车站点 | 用户忘记操作 → 订单一直 IN_PROGRESS |
| 6 | 电子围栏禁止乱停 | 依赖用户手动选择还车站点 + 确认 | 无法物理阻止乱停 |
| 7 | 真实支付 | 模拟支付（数据库操作） | 无真实资金流 |
| 8 | 押金预授权 | 无 | 无法约束用户行为 |

## 5. 与 Sprint 4 计划的对齐

上述冲突分析直接决定了 Sprint 4 的技术路线：

1. **保持 Station 模式**（第 2.1 节）：Station 是唯一可靠的物理锚点
2. **不做自由流动**（第 3 节第 1 项）：已否决
3. **计费保持预订锁定模式**（第 2.5 节）：不做按实际时长动态计价
4. **不模拟 IoT 数据**（第 2.3 节）：不新增 `batteryLevel`、`lastPositionUpdate` 等假字段
5. **支付模拟但流程完整**（第 2.5 节）：有 PaymentModal、有成功/失败处理，只是不接真实网关

---

## 6. 给后续读者的说明

如果你将来要在本项目中加入硬件层（哪怕是模拟硬件），以下是最优先的切入点：

1. **MQTT Broker**（如 EMQX）+ 设备主题设计 → 解决设备通信
2. **Scooter 模型增加 `batteryLevel`、`firmwareVersion`、`lastHeartbeat`** → 模拟设备状态
3. **QR 码生成 + 摄像头扫描（需要 HTTPS + 移动端）** → 解锁验证
4. **WebSocket 推送** → 实时状态同步（替代轮询）
5. **按实际骑行时长计费** → 用 `actualStartTime` → `actualEndTime` 差值重新计算费用
