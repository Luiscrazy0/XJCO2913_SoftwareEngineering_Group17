# Sprint 1 - 阶段3：车辆发现页（ScooterListPage）实现指南

## 一、核心需求分析（第一性原理）

### 1.1 用户核心需求
- 查看当前可用的电动滑板车列表
- 了解车辆的基本信息（位置、状态）
- 能够选择车辆进行预约

### 1.2 系统约束条件
- 后端API：GET /scooters 返回所有车辆
- 数据库模型：Scooter { id, location, status }
- 前端类型定义：Scooter接口已存在
- 状态过滤：前端需要过滤AVAILABLE状态的车辆

## 二、实际数据结构

### 2.1 后端返回数据（Prisma模型）
```typescript
// 实际数据库模型
interface Scooter {
  id: string           // UUID
  location: string     // 位置描述（字符串）
  status: ScooterStatus // 'AVAILABLE' | 'UNAVAILABLE'
}
```

### 2.2 前端类型定义（当前状态）
```typescript
// 前端types/index.ts中的定义（需要更新）
interface Scooter {
  id: string
    latitude: number
    longitude: number
    address: string
  }
  createdAt: string
  updatedAt: string
}
```

## 三、实现路径规划

### 3.1 第一步：对齐数据类型
**问题**：前端类型定义与后端实际数据结构不匹配
**解决方案**：
1. 更新前端Scooter接口，匹配后端实际字段
2. 或者创建适配器函数转换数据格式

### 3.2 第二步：实现API集成
**当前状态**：scooters.ts API封装已存在，但需要验证
**验证步骤**：
1. 测试GET /scooters接口是否正常工作
2. 确认返回数据格式
3. 处理可能的错误状态

### 3.3 第三步：构建页面组件
**核心组件结构**：
```
ScooterListPage
├── 页面容器（布局、样式）
├── 数据加载状态管理
│   ├── Loading状态
│   ├── Error状态
│   ├── Empty状态
│   └── Success状态
├── 车辆统计信息
├── 车辆网格列表
│   └── ScooterCard组件
└── 预约触发逻辑
```

### 3.4 第四步：状态过滤与显示
**过滤逻辑**：
- 从API获取所有车辆
- 前端过滤status === 'AVAILABLE'的车辆
- 显示可用车辆数量统计

## 四、具体实现步骤

## 六、风险与应对

### 6.1 数据不一致风险
**风险**：前后端数据类型不匹配导致显示错误
**应对**：优先采用方案A（更新前端类型），确保数据一致性

### 6.2 API兼容性风险
**风险**：后端API变更影响前端功能
**应对**：添加API响应验证，使用TypeScript类型保护

### 6.3 状态管理风险
**风险**：复杂的状态逻辑导致bug
**应对**：使用TanStack Query简化数据获取和状态管理

## 附录：关键决策记录

### 决策1：数据类型对齐
**选择**：更新前端Scooter接口，匹配后端实际数据结构
**理由**：保持数据一致性，减少转换逻辑，降低复杂度

### 决策2：状态过滤位置
**选择**：前端过滤AVAILABLE状态车辆
**理由**：后端API未提供状态过滤参数，前端过滤简单有效

### 决策3：预约流程触发
**选择**：预约按钮点击触发事件，具体预约逻辑在阶段4实现
**理由**：关注点分离，本阶段专注于车辆发现功能

### 决策4：错误处理策略
**选择**：使用TanStack Query内置错误处理，配合自定义UI反馈
**理由**：利用成熟解决方案，减少自定义错误处理代码
