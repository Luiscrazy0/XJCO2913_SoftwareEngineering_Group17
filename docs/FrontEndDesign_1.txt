# Sprint 1 前端 MVP 交互设计方案

Sprint 1 的目标是实现系统的最小可行产品（MVP），重点验证核心业务流程：

用户注册登录 → 浏览车辆 → 发起预约 → 系统创建订单。

本阶段不追求完整功能或复杂 UI，而是优先保证系统可以完成一次完整租车流程。

前端建议采用以下技术栈：

React（Vite）
Tailwind CSS
Shadcn UI（或 Ant Design 二选一）
TanStack Query（React Query）

其中 UI 组件库建议 **只使用一种**，避免样式冲突。

---

# 1 核心用户路径

Sprint 1 实现两类基础用户路径。

## Customer（租车用户）

```
Register / Login
        ↓
Browse Scooters
        ↓
Create Booking
        ↓
My Bookings
```

用户无需进入单独的车辆详情页，可直接在车辆列表中完成预约。

---

## Manager（管理员）

管理员功能仅提供最基础能力：

```
Login
   ↓
Fleet Management
   ↓
Add Scooter
```

管理员界面主要用于系统测试和演示，不属于核心租车流程。

---

# 2 页面交互设计

Sprint 1 控制页面数量，避免复杂 UI 结构。

共 **4 个页面**：

Auth
Scooter List
My Bookings
Admin Fleet

预约操作通过 **Booking Modal** 完成，而不是单独页面。

---

# 页面 A：身份认证页 (Auth Page)

对应 API

```
POST /auth/register
POST /auth/login
```

功能说明

Auth 页面提供登录与注册两种功能，通过 Tab 切换。

UI 结构

```
Card
 ├── Tabs
 │     ├── Login
 │     └── Register
 └── Form
```

表单字段

Login

```
email
password
```

Register

```
email
password
confirm password
```

交互逻辑

前端进行基础校验：

* 邮箱格式
* 密码长度
* 两次密码一致

登录成功后：

```
保存 token
保存 user 信息
跳转首页
```

推荐返回结构：

```json
{
  "token": "...",
  "user": {
    "id": 1,
    "email": "user@email.com",
    "role": "CUSTOMER"
  }
}
```

前端根据 `user.role` 判断跳转：

```
CUSTOMER → /scooters
MANAGER → /admin
```

Token 存储建议：

```
localStorage
```

# 页面 C：预约弹窗 (Booking Modal)

对应 API

```
POST /bookings
```

该组件负责创建预约。

UI 结构

```
Modal
   |
Scooter Info
   |
Booking Form
   |
Confirm Booking
```

表单字段

```
HireType
StartTime
```

HireType 使用按钮选择：

```
1 Hour
4 Hours
1 Day
1 Week
```

交互逻辑

用户选择：

```
HireType
StartTime
```

提交请求：

```
POST /bookings
```

请求结构：

```json
{
  "scooterId": 3,
  "hireType": "HOUR_1",
  "startTime": "2026-03-20T10:00:00"
}
```

成功返回：

```json
{
  "bookingId": 12,
  "totalCost": 12.5
}
```

前端展示：

```
Booking success
```

然后跳转：

```
/my-bookings
```

注意：

Sprint 1 **价格计算由后端完成**，前端仅展示返回结果。

同时，为防止同一车辆在同一时间被重复预约：

* 前端只负责发起请求
* **后端负责时间冲突校验**
* 数据库负责最终一致性约束

# 5 Token 生命周期管理

从安全设计角度，Token 必须包含完整生命周期：

```
Login → Use → Logout
```

实现方式：

Login

```
保存 token 到 localStorage
```

API 请求

```
Authorization: Bearer <token>
```

Logout

```
localStorage.removeItem("token")
清除 user 状态
跳转 /login
```

同时前端应实现 **Auth Guard**：

若用户未登录：

```
自动跳转 /login
```

# 7 Sprint 1 完成标准

Sprint 1 不以 UI 完整度为目标，而以 **业务流程可运行** 为标准。

系统应能够完成：

```
1 用户注册
2 用户登录
3 浏览车辆列表
4 创建预约
5 在数据库中生成 booking
6 用户查看自己的 booking
7 用户能够安全退出（Logout）
```

只要上述流程完整可运行，即可认为 **Sprint 1 MVP 完成**。
