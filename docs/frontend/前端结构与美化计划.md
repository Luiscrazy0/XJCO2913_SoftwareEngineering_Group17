# 前端结构与美化计划（Scooter Sharing Platform）

本文目标：
- 让你快速看懂当前前端（React + Vite + TS + Tailwind）的**结构与数据流**
- 基于现状给出一份**可落地的前端美化/重构路线图**（按优先级拆分）

## 1. 技术栈与入口

- 构建工具：Vite（`frontend/vite.config.ts`）
- UI：React 19（`frontend/src/main.tsx` -> `frontend/src/App.tsx`）
- 路由：React Router（`frontend/src/router/AppRouter.tsx`）
- 数据请求：Axios（`frontend/src/utils/axiosClient.ts`）
- 服务端状态：TanStack React Query（`frontend/src/utils/queryClient.ts`）
- 样式：Tailwind CSS v4（`frontend/src/index.css` 顶部 `@import "tailwindcss";`）

应用挂载链路：
`frontend/src/main.tsx` → 渲染 `<App />`
`frontend/src/App.tsx` → 依次挂 Provider：`QueryClientProvider` → `ToastProvider` → `AuthProvider` → `AppRouter`

## 2. 当前目录结构（按“你会改哪里”来解释）

`frontend/src/`
- `api/`：按领域划分的 API 调用（`auth.ts` / `scooters.ts` / `bookings.ts`），统一走 `utils/axiosClient.ts`
- `components/`：可复用 UI 组件（卡片、弹窗、Navbar、空态/错态、Toast 等）
- `components/admin/`：后台专用组件（表格、统计卡、确认弹窗等）
- `context/`：全局状态（目前只有 `AuthContext.tsx`）
- `pages/`：页面（路由入口组件）：`AuthPage` / `ScooterListPage` / `MyBookingsPage` / `AdminFleetPage` / `ForbiddenPage`
- `router/`：路由表 + 受保护路由包装（`AppRouter.tsx`）
- `types/`：前端领域类型（与后端 DTO/枚举对齐）
- `utils/`：基础设施（axios 实例、queryClient 等）
- `index.css`：全局样式 + CSS Token（目前混有模板遗留样式）

## 3. 关键运行逻辑（你需要先看懂的 3 条“主干”）

### 3.1 登录态与鉴权（Auth）

- 状态源：`localStorage`（`auth_token` + `user`）
- 管理位置：`frontend/src/context/AuthContext.tsx`
  - `login/register`：调用 `authApi`，解析 JWT，写入 localStorage，并根据角色跳转页面
  - `logout`：清空 localStorage，发 Toast，并跳转到登录页
- 请求侧：`frontend/src/utils/axiosClient.ts`
  - request interceptor：读取 `auth_token` 注入 `Authorization: Bearer ...`
  - response interceptor：遇到 `401` 派发 `auth:session-expired`，由 `AuthContext` 监听并 logout

### 3.2 路由与权限（Router + ProtectedRoute）

路由表在：`frontend/src/router/AppRouter.tsx`
- 公共：`/` `/auth` `/403` `/test-scooters`
- 受保护：`/scooters` `/bookings`
- 管理员：`/admin`（`requiredRole="MANAGER"`）

权限门禁在：`frontend/src/components/ProtectedRoute.tsx`
- 未登录：记下 `redirect_path`，跳回 `/`
- 角色不符：Toast 提示 + 跳 `/403`

### 3.3 数据获取（React Query + api）

典型模式：
- Page 用 `useQuery/useMutation` 调用 `api/*`
- 成功后 `invalidateQueries` 刷新列表（例如 `AdminFleetPage` / `BookingModal`）

建议你读代码的顺序：
1) `frontend/src/App.tsx`
2) `frontend/src/router/AppRouter.tsx`
3) `frontend/src/context/AuthContext.tsx` + `frontend/src/utils/axiosClient.ts`
4) `frontend/src/pages/*`（按用户路径：Auth → Scooters → BookingModal → MyBookings；管理员再看 Admin）

## 4. 现状 UI 评估（为什么“能用但不美”）

1) **样式体系不统一**
   - 大部分页面/组件用 Tailwind class
   - `AuthPage` 大量使用 inline style + 少量 class（并通过 `index.css` 打补丁）

2) **全局 CSS 有模板遗留**
   - `frontend/src/index.css` 里 `#root` 固定宽度、`text-align:center`、边框等更像 Vite 模板，而不是应用级布局
   - `frontend/src/App.css` 是模板残留，且目前并未被引用（容易误导后续维护）

3) **交互/反馈不统一**
   - 既有 `ToastProvider`（后台/权限使用），也有页面内“自制 notification”（`MyBookingsPage`）
   - 一些页面用 `window.location.href` 导航，会导致整页刷新（体验与状态保持较差）

4) **视觉语言不统一**
   - 颜色：蓝/绿/emerald 混用
   - 圆角、阴影、边框、按钮风格在不同页面差异明显

## 5. 美化路线图（按优先级拆分，保证“每步都有收益”）

### P0（1 天内）：不改业务，先统一“底座”

- 统一全局布局：清理 `frontend/src/index.css` 的模板 `#root` 样式，改为“页面自己决定容器宽度”
- 统一主题 Token：保留 `:root` 的颜色变量，但把它们整理成「基础色/语义色/组件色」三层
- 统一导航方式：逐步替换 `window.location.href` → `react-router` 导航（避免整页刷新）
- 统一反馈：把 `MyBookingsPage` 的通知改成用 `ToastProvider`
- 统一 API base URL：把 `axiosClient` 的 `BASE_URL` 改为读取 `VITE_API_BASE_URL`（默认 `http://localhost:3000`）

交付标准：页面不“变丑”，但整体观感一致，跳转更顺滑，提示统一。

### P1（2–3 天）：做一套最小 Design System（组件原子化）

目标：减少 Tailwind 复制粘贴，让风格能“批量升级”。

- 建立 `components/ui/`：
  - `Button`（primary/secondary/danger/ghost + loading + disabled）
  - `Input`（label、hint、error、left/right icon）
  - `Card` / `Section` / `PageHeader`
  - `Modal`（可复用：BookingModal、AddScooterModal、DeleteConfirmationModal）
  - `Badge` / `Alert` / `Skeleton`
- 把重复样式从页面里抽出来：先改 `AdminFleetPage`（重复多、收益最大）

交付标准：新增页面不需要“现场拼 class”；风格自动一致。

### P2（3–5 天）：页面级美化（体验 + 信息层级）

- `AuthPage`：
  - 改为 Tailwind + ui 组件（去掉 inline styles）
  - 更清晰的错误提示/校验提示，表单可访问性（label/aria）完善
- `ScooterListPage`：
  - 增加筛选/搜索（位置关键字、状态），列表头信息更清晰
  - 把 DEV 调试块改成可折叠 Debug 面板（或受 `VITE_DEBUG` 控制）
- `MyBookingsPage`：
  - 状态分组（待支付/已确认/已完成/已取消），增强信息密度与可读性
  - 操作按钮风格统一（取消/支付的危险与主按钮语义）
- `AdminFleetPage`：
  - 表格：固定表头、空态、批量操作（可选）
  - 将统计卡/表格/弹窗统一到同一视觉语言（同色系、同圆角、同阴影）

### P3（可选）：高级体验（看你们课程/展示需求）

- 深色模式（利用现有 `prefers-color-scheme` token）
- 动效与过渡（弹窗、toast、列表刷新）
- 可访问性（Modal focus trap、键盘操作）
- 图标体系：复用 `frontend/public/icons.svg`（而不是每个组件都内联 SVG）

## 6. 建议的“先改哪里”（最短路径）

如果目标是“尽快变好看且不踩坑”，建议顺序：
1) 清理 `frontend/src/index.css` 里模板 `#root` 样式（P0）
2) 引入 `VITE_API_BASE_URL` 配置（P0）
3) 把 `MyBookingsPage` 通知改用 `ToastProvider`（P0）
4) 先做 `components/ui/Button` + `Input` + `Card`（P1）
5) 重写 `AuthPage`（P2，观感提升最大）
